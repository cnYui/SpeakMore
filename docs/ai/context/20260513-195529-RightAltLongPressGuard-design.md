# RightAlt 长按拦截提示设计

## 背景

当前 `RightAlt` 作为语音输入主快捷键，但前端对 `global-keyboard` 事件的消费过于直接：

- 只要看到 `RightAlt` 处于按下态，就可能立即触发识别
- 长按 `RightAlt` 时，按键状态会在当前实现里被重复消费
- 结果是用户还没开始说话，就已经进入转写或转译

这不是后端识别问题，而是快捷键判定缺少：

- 单次消费
- 长按拦截
- 用户可见反馈

## 目标

1. 长按 `RightAlt` 超过 `500ms` 时，不进入识别。
2. 长按后弹出提示弹窗，明确告诉用户需要短按。
3. 弹窗样式与项目当前设置页的快捷键区域保持一致。
4. 弹窗显示期间：
   - 松开 `RightAlt` 不自动关闭
   - 用户可以点右上角关闭
   - 用户再次点按 `RightAlt` 时，先关闭弹窗，再允许这次按键正常进入识别
   - 如果再次变成长按，则重新拦截并重新展示弹窗
5. 修正快捷键消费逻辑，避免一个按下周期内被重复识别。

## 非目标

1. 不修改后端识别协议。
2. 不改动现有语音状态机语义。
3. 不在这次处理里补完麦克风选择、历史统计等无关问题。

## 用户交互

### 正常短按

1. 用户按下 `RightAlt`
2. 系统进入等待判定状态
3. 若在 `500ms` 内松开，视为正常快捷键
4. 触发对应识别流程，并显示现有悬浮胶囊

### 长按拦截

1. 用户按下 `RightAlt`
2. 持续按住超过 `500ms`
3. 当前这次按键被标记为 `blocked`
4. 不进入识别
5. 展示提示弹窗

### 弹窗显示时的再次点按

1. 弹窗已显示
2. 用户再次点按 `RightAlt`
3. 先关闭弹窗
4. 这次新的点按继续按正常快捷键流程处理
5. 如果这次再次长按超过 `500ms`，则再次拦截并重新展示弹窗

### 弹窗关闭方式

1. 点击右上角 `×`
2. 再次点按 `RightAlt`

松开 `RightAlt` 不会自动关闭弹窗。

## 提示文案

默认文案：

- 标题：`检测到长按快捷键`
- 说明：`长按 Right Alt 不会开始语音输入。请短按 Right Alt，或使用 Right Alt + Space / Right Alt + Right Shift。`

## 视觉设计

复用当前项目设置页中的视觉语言：

- 白底
- 细灰边框
- 12px~16px 圆角
- 黑色主文案，灰色辅助文案
- 快捷键使用与设置页一致的 chip 形式
- 右上角使用关闭按钮

呈现形式为主窗口内浮层，不修改系统级原生弹窗。

原因：

- 当前项目已经使用 MUI，自定义浮层成本低
- 样式能与设置页快捷键区域统一
- 不引入额外原生窗口状态同步问题

## 方案选择

### 方案 A：只增加启动延迟

优点：

- 改动小

缺点：

- 不能真正解决长按重复触发
- 只是把误触时间推迟

### 方案 B：在 Dashboard 内局部补长按提示

优点：

- 实现快

缺点：

- 快捷键逻辑仍然绑在 Home 页
- 切页后行为继续不稳定

### 方案 C：把快捷键守卫上提到 AppShell 级

优点：

- 统一处理 `RightAlt` 判定
- 同时解决长按误触和页面耦合问题
- 后续扩展其它快捷键规则更清晰

缺点：

- 需要改动 `AppShell`、`Dashboard` 和测试

本次采用方案 C。

## 状态设计

新增一个独立于 `recorder` 的快捷键守卫状态：

- `idle`：无按键周期
- `pressing`：`RightAlt` 已按下，等待短按/长按判定
- `blocked`：当前按键周期已因长按被拦截
- `modalVisible`：提示弹窗可见

补充状态字段：

- `pressStartedAt`
- `longPressTimer`
- `blockedPressId`
- `isModalVisible`

核心约束：

1. 一个按下周期最多只产生一次结果：
   - 要么正常识别
   - 要么长按拦截
2. 一旦进入 `blocked`，直到本次按键周期结束前，不得进入识别。
3. 弹窗显隐与按键周期解耦：
   - `blocked` 可以结束
   - `modalVisible` 仍可保留，直到用户主动关闭或再次点按关闭

## 数据流

1. 主进程继续通过 `global-keyboard` 向 renderer 发送键盘状态。
2. `AppShell` 层监听 `global-keyboard`。
3. 快捷键守卫根据 `RightAlt / Space / RightShift` 状态推进自己的状态机。
4. 守卫决定：
   - 是否关闭弹窗
   - 是否展示弹窗
   - 是否把本次事件转给 `toggleRecording`
5. `Dashboard` 不再直接监听 `global-keyboard`，只负责展示语音状态。

## 文件变更

### 新增

- `electron-app/renderer/src/services/shortcutGuard.ts`
  - 负责长按判定、单次消费、弹窗状态、事件解释

### 修改

- `electron-app/renderer/src/components/AppShell.tsx`
  - 承接全局快捷键监听
  - 渲染长按提示弹层
- `electron-app/renderer/src/pages/Dashboard.tsx`
  - 去掉直接消费 `global-keyboard` 的逻辑
  - 保留语音会话展示
- `electron-app/renderer/src/uiTokens.ts`
  - 如有必要，抽出弹层/快捷键 chip 共用样式
- `electron-app/renderer/ui-structure.test.mjs`
  - 增加长按拦截和弹窗存在性的结构断言

## 错误兜底

1. 如果 `ipcRenderer` 不可用：
   - 快捷键守卫注册为空行为
   - 弹窗逻辑不报错
2. 如果长按定时器未正常清理：
   - 在 `AppShell` 卸载时统一清理
3. 如果弹窗关闭时仍有旧定时器：
   - 关闭动作同时重置守卫内部状态
4. 如果收到异常键盘 payload：
   - 守卫只消费可识别的 `RightAlt / Space / RightShift`
   - 其它输入直接忽略

## 测试策略

### 结构测试

在 `ui-structure.test.mjs` 中至少覆盖：

1. `AppShell` 不再仅由 `Dashboard` 决定快捷键行为
2. 存在 `500ms` 长按阈值
3. 存在提示弹窗标题或文案
4. 弹窗包含关闭入口
5. 长按拦截与正常识别不是同一条直接逻辑分支

### 人工验证

1. 短按 `RightAlt`：
   - 正常开始语音输入
2. 长按 `RightAlt` 超过 `500ms`：
   - 不进入识别
   - 弹出提示
3. 弹窗显示后松开：
   - 弹窗继续保留
4. 点击 `×`：
   - 弹窗关闭
5. 弹窗显示后再次短按 `RightAlt`：
   - 先关闭弹窗
   - 正常进入识别
6. 弹窗显示后再次长按 `RightAlt`：
   - 先关闭旧弹窗
   - 不进入识别
   - 重新弹出提示

## 风险

1. 如果快捷键守卫仍留在页面级，会继续出现切页失效问题。
2. 如果只拦截 `RightAlt`，但未处理组合键状态复位，可能出现 `Space/RightShift` 残留。
3. 如果弹窗状态和按键周期耦合过深，容易出现关闭后下一次按键不触发。

## 结论

这次应把问题定义为“快捷键消费治理”，不是“识别速度调慢”。  
真正要补的是：

- 长按拦截
- 单次消费
- 可见提示
- 页面级解耦

按这个方案实现后，`RightAlt` 的误触会从入口被阻断，而不是在识别已开始后再补救。
