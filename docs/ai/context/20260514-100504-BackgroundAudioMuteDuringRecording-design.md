# 录音期间后台音频静音设计

## 背景

当前项目的语音输入交互已经固定为：

- 点按 `RightAlt` 开始录音
- 再次点按 `RightAlt` 结束录音

这次需求不是把交互改成 `PTT`，也不是要求系统级“暂停所有声音”，而是：

- 保持现有点按式录音模型不变
- 在录音期间自动静音除 Typeless 自己外的后台声音
- 在本轮录音结束后恢复这次被 Typeless 主动静音的音频会话

## 目标

1. 保持当前 `RightAlt` 边沿触发与长按保护语义不变。
2. 仅在 Windows 上启用“录音期间静音后台音频”。
3. 只静音当前活跃的后台音频会话，排除 Typeless 自己。
4. 结束录音后只恢复这次被 Typeless 改动过的会话。
5. 任意失败路径都必须兜底恢复，不能把外部应用遗留在静音状态。

## 非目标

1. 不改成 `PTT`。
2. 不实现真正的“暂停播放进度”。
3. 不在这次加入应用白名单、黑名单或设置页开关。
4. 不扩展到 macOS / Linux。
5. 不处理录音开始后才新建出来的后台音频会话。

## 需求澄清

### 为什么不是“暂停所有声音”

Windows 能稳定控制的是音频会话的静音状态，不是统一的跨应用暂停状态。  
对第三方应用做“全局暂停/恢复”既不可靠，也无法保证恢复语义正确。

### 正确的问题定义

真正要解决的是：

- 用户说话时不要被后台声音干扰

所以本次采用：

- 录音期间静音后台活跃音频会话
- 录音结束后按快照恢复

## 方案选择

### 方案 A：发送系统媒体播放/暂停键

优点：

- 实现表面上简单

缺点：

- 不能保证所有应用响应
- 无法精确排除 Typeless
- 无法保证“只恢复本次被改动的对象”

不采用。

### 方案 B：引入 Electron 原生音频控制模块

优点：

- 主进程调用更直接

缺点：

- 需要新增原生依赖
- 与 Electron 版本兼容、构建和维护成本更高

当前项目不值得为这一点能力扩展原生依赖链路，不采用。

### 方案 C：PowerShell + 内嵌 C# + Windows Core Audio API

优点：

- 贴合当前项目已有的 Windows 脚本调用方式
- 不需要新增 Node 原生依赖
- 可以按音频会话枚举、过滤进程、读取和设置静音状态
- 便于把恢复语义做对

缺点：

- 仅适用于 Windows
- 会引入一段 COM 互操作脚本

本次采用方案 C。

## 架构设计

### 总体原则

- 系统音频控制只放在 `electron-app/main.js`
- `renderer` 只负责在录音生命周期节点调用主进程 IPC
- 恢复快照由主进程保管，不能留在一次性脚本进程里

### 组件划分

#### 1. `electron-app/audio-session-control.ps1`

职责：

- 通过 PowerShell 内嵌 C# 调用 Windows Core Audio COM 接口
- 枚举默认输出设备上的活跃音频会话
- 读取每个会话的进程 ID、状态、静音状态
- 执行两类动作：
  - `mute-active-sessions`
  - `restore-sessions`
- 输出 JSON 结果给主进程

#### 2. `electron-app/main.js`

职责：

- 维护后台音频静音控制器
- 记录本轮录音的静音快照
- 暴露 IPC：
  - `audio:mute-background-sessions`
  - `audio:restore-background-sessions`
- 在应用退出时兜底恢复

#### 3. `electron-app/renderer/src/services/recorder.ts`

职责：

- 在录音真正开始后请求主进程静音后台会话
- 在录音完成、失败、销毁时请求主进程恢复
- 不改变现有录音状态机和语音协议

## 数据与状态

主进程新增一组运行时状态：

- `backgroundMuteActive: boolean`
- `mutedBackgroundSessions: array`

其中 `mutedBackgroundSessions` 保存本轮被主动静音的会话快照。  
每个快照至少需要包含：

- 会话标识信息
- 进程 ID
- 恢复所需的原始静音状态

核心约束：

1. 同一时间只允许存在一份活跃快照。
2. 新一轮静音前，如果旧快照未清理，先尝试恢复。
3. 恢复只针对本轮被 Typeless 主动静音的会话。

## 录音生命周期接线

### 开始录音

`recorder.ts` 中的 `startRecording()` 保持现有链路：

1. 建立 WebSocket
2. 获取麦克风
3. `mediaRecorder.start()`
4. 状态进入 `recording`
5. 异步调用 `audio:mute-background-sessions`

原因：

- 录音起始不能被系统音频控制阻塞
- 静音应尽量快，但不应成为录音启动的前置硬依赖

### 结束录音

`stopRecording()` 不在点击结束当下立即恢复后台音频。  
恢复时机放在本轮录音生命周期真正结束时：

- `completed`
- `error`
- `disposeRecorder`

原因：

- 用户目标是“说话这段时间不要被背景音干扰”
- 当前项目停止录音后还会经历转写、润色、粘贴
- 如果点击停止就立即恢复，体验会突兀回声

## 会话过滤规则

静音时只处理满足下面条件的会话：

1. 会话状态为 `Active`
2. 会话所属进程不属于 Typeless 自己
3. 开始静音前该会话不是静音状态

恢复时只处理：

1. 本轮被 Typeless 主动静音过的会话
2. 恢复为原始静音状态

这保证：

- 本来就静音的应用保持静音
- Typeless 不会误恢复其他应用的用户手动设置

## 错误处理

### 录音开始前失败

如果静音还没发生，不恢复。  
如果静音已经发生，立即恢复后再进入错误态。

### 录音中失败

统一由 `failSession()` 触发恢复。

### 转写或粘贴阶段失败

这些仍属于本轮录音生命周期的一部分，进入 `error` 时必须恢复。

### 退出应用

主进程在 `before-quit` 或等价退出路径再次尝试恢复，作为最终兜底。

### 脚本失败

如果 PowerShell / COM 调用失败：

- 不阻塞现有录音流程
- 记录错误日志
- 让语音输入继续可用

本次功能优先级低于录音主链路稳定性。

## 测试策略

### 结构测试

至少覆盖：

1. `main.js` 存在后台音频静音脚本入口与新 IPC。
2. `recorder.ts` 在开始录音后调用静音 IPC。
3. `recorder.ts` 在完成、失败、销毁路径调用恢复 IPC。
4. 旧的 `audio:mute` / `audio:unmute` 空壳不再是唯一入口。

### 人工验证

在 Windows 上验证：

1. 后台应用正在播放音频时开始录音，后台音频被静音。
2. Typeless 自己不被错误静音。
3. 正常完成录音后，后台音频恢复。
4. WebSocket / 后端异常后，后台音频仍恢复。
5. 录音开始前本来就静音的应用，结束后仍保持静音。

## 风险与边界

1. Windows 控制粒度是音频会话，不一定等同于“一个应用一个对象”。
2. 只处理录音开始时已经活跃的会话；录音期间新出现的会话不在第一版范围内。
3. PowerShell + COM 方案会增加一些 Windows 专用代码，但比新增 Electron 原生依赖更稳妥。

## 结论

本次功能应定义为：

- 保持现有点按式录音模型
- 通过 Windows 音频会话接口静音后台活跃会话
- 由主进程统一维护静音快照和恢复语义
- 任意结束路径都兜底恢复

这样能解决真实问题，又不把项目错误地改造成 `PTT` 或不可靠的“全局暂停”。
